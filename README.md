# Loblaw Bio ŌĆō Clinical Trial Immune Analysis

This repo contains a reproducible SQLite-backed data pipeline and an interactive dashboard for analyzing immune cell populations from `cell-count.csv`
---

## How to run (Codespaces or local)

From the repository root:

```bash
make setup
make pipeline
make dashboard
```

### What each Make target does

- `make setup`  
  Installs dependencies from `requirements.txt`.

- `make pipeline`  
  Runs the full pipeline end-to-end with no manual steps:
  1. Creates/initializes the SQLite database (`loblaw.db`) and loads `cell-count.csv` (Part 1)
  2. Generates the per-sample frequency summary table (Part 2)
  3. Runs responder vs non-responder analysis + saves stats + generates plots (Part 3)
  4. Runs baseline subset queries and writes summary counts (Part 4)

- `make dashboard`  
  Starts the local Streamlit server for the interactive dashboard.

### Outputs generated by the pipeline

Running `make pipeline` produces:

- `loblaw.db` (SQLite database)
- `analysis/summary_frequencies.csv` (Part 2 output)
- `analysis/responder_vs_nonresponder_stats.csv` (Part 3 stats output)
- `analysis/boxplot_percentages.png` (Part 3 visualization)
- `analysis/part4_baseline_miraclib_pbmc_samples.csv` (Part 4 subset output)
- `analysis/part4_counts.txt` (Part 4 counts summary)

If you want to validate a clean run from scratch:

```bash
rm -f loblaw.db
rm -rf analysis
make pipeline
```

## Relational database schema

The schema is normalized to keep metadata separate from cell count measurements and to scale cleanly as the dataset grows

### `samples` (one row per sample)

Primary key: `sample_id` (from CSV column `sample`)

Columns (aligned to the CSV):
- `sample_id`
- `project`
- `subject_id` (from `subject`)
- `condition`
- `age`
- `sex`
- `treatment`
- `response` (nullable, since some samples can be missing response)
- `sample_type`
- `time_from_treatment_start`

### `cell_counts` (long-format counts)

Primary key: (`sample_id`, `population`)

Columns:
- `sample_id` (FK ŌåÆ `samples.sample_id`)
- `population` (e.g., `b_cell`, `cd8_t_cell`, `cd4_t_cell`, `nk_cell`, `monocyte`)
- `count`

### Rationale and scalability

The immune cell counts are stored in long format (`sample_id`, `population`, `count`) instead of ŌĆ£wideŌĆØ columns inside the database. This makes the design scalable and flexible:

- If new immune populations get added later, the schema does not change; we just insert new `(sample_id, population)` rows.
- Analytics (grouping, filtering, comparisons) stay consistent across projects and populations.
- Sample metadata stays in one place (`samples`), which keeps queries and joins predictable.

Indexes are added on commonly-filtered fields (`condition`, `treatment`, `sample_type`, `response`, `sex`, `time_from_treatment_start`) so queries remain fast when scaling to hundreds of projects and thousands of samples.

## Code structure

- `load_data.py`  
  Single responsibility: initialize the database schema and load the CSV into normalized tables. This is separated so Part 1 can be rerun independently and so the pipeline has a clear ŌĆ£data ingestionŌĆØ step.

- `pipeline.py`  
  Runs Parts 2ŌĆō4 and writes all required outputs to `analysis/`. Keeping this as a single script makes the grading workflow simple: one command (`make pipeline`) generates everything.

- `dashboard.py`  
  Streamlit app that reads the generated outputs and provides an interactive view (tables + plot). This stays separate from the pipeline so dashboard startup is fast and does not rerun analysis

- `Makefile`  
  Provides three required entry points (`setup`, `pipeline`, `dashboard`) so the project runs predictably in Codespaces.
